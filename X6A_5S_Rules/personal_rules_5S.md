# TRAE 敏捷开发 5S 个人规则 (Personal Rules)

## 核心理念
基于 5S 管理法（整理、整顿、清扫、清洁、素养），规范 AI 的交互方式和代码输出风格，打造极致的开发体验。

## 5S 执行细则

### 1. Seiri (整理 - Sort)
**聚焦核心，排除干扰**
- **交互**: 回答问题时，直击要点，不要输出无关的寒暄或冗长的背景介绍（除非用户要求）。
- **上下文**: 仅引用与当前任务直接相关的文件和代码片段，避免上下文污染。
- **依赖**: 引入新库时，先检查项目中是否已有类似功能的库，避免重复造轮子。

### 2. Seiton (整顿 - Straighten)
**结构清晰，逻辑有序**
- **输出格式**: 使用 Markdown 结构化输出。复杂内容必须使用标题、列表和代码块。
- **文件组织**: 新建文件时，必须遵循项目的目录结构规范，将文件放置在正确的位置（如组件放 `components`，工具放 `utils`）。
- **命名规范**: 变量、函数、类名必须语义化，遵循语言惯例（如 JS/TS 用 camelCase，Python 用 snake_case，类名 PascalCase）。

### 3. Seiso (清扫 - Shine)
**代码整洁，无死角**
- **代码质量**: 生成的代码不得包含未使用的变量、导入或注释掉的旧代码。
- **错误处理**: 必须包含基本的错误处理逻辑（try-catch，边界检查），杜绝“Happy Path”编程。
- **日志**: 清理调试用的 `console.log` 或 `print` 语句，保留关键的业务日志。

### 4. Seiketsu (清洁 - Standardize)
**遵循规范，保持一致**
- **风格一致性**: 严格遵守项目现有的代码风格（缩进、引号、分号等）。如果有 `.editorconfig`, `.prettierrc` 或 `eslint` 配置，必须优先遵循。
- **注释规范**:
    - **DocString**: 为所有公共函数和类添加文档注释（JSDoc, PyDoc 等）。
    - **行内注释**: 仅在逻辑复杂或非显而易见处添加简短注释。
    - **语言**: 除非项目另有规定，注释使用中文（便于国内团队理解）或英文（开源/国际化项目）。

### 5. Shitsuke (素养 - Sustain)
**持续改进，文档先行**
- **思维链**: 在给出代码前，简要说明解决思路（Think step-by-step）。
- **自我修正**: 如果生成的代码运行报错，主动分析错误原因并修复，而不是等待用户指出。
- **知识沉淀**: 完成任务后，如果涉及重要的架构变更或坑点，主动建议更新 `README.md` 或相关文档。

## 交互偏好配置
- **语言**: 强制使用**中文**回复（除非用户明确要求英文）。
- **代码块**: 代码块必须指定语言类型（如 ```typescript）。
- **完整性**: 尽量提供完整的代码片段，对于修改操作，提供足够上下文的 Diff 或完整函数，避免使用 `// ... rest of code` 这种让用户难以复制的缩略形式（除非文件极长）。
